Hackathon Details:

Build a production-style form builder SaaS where users can create dynamic forms, publish public or unlisted form links and collect responses.

You will be given a Turborepo starter repository. Your job is to extend it into a complete Typeform-style product with form creation, public form filling, analytics, email flows and API documentation.

Creators should be able to create forms, add fields, configure validations, choose themes, publish/unpublish forms and share links with respondents. Public users should be able to fill forms and submit responses without logging in.

Forms should support dynamic schemas with configurable fields, validation rules and required/optional settings. Use Zod for schema validation and tRPC for type-safe APIs. The project must use Turborepo, tRPC, Zod, Drizzle ORM and Scalar as core parts of the stack.

The product should also feel like a real SaaS application with:

landing page

pricing page

demo-ready deployment

API documentation

sample seeded data

demo credentials

The demo should already contain sample forms, themes, responses and analytics so judges can review the product quickly.

Forms should support creative themes around movies, anime, games, startups, tech companies, operating systems, events or communities.

Some FAQs:

The project must use Turborepo, tRPC, Zod, Drizzle ORM and Scalar. You are free to use Next.js (or) React.js for frontend. Public forms are published forms that can appear in public-facing areas of the product, such as explore pages, featured forms or template galleries. Anyone can open and submit them. Unlisted forms are also published, but they should not appear anywhere publicly inside the product. Only people with the direct form link should be able to open and submit them. Yes. Both public and unlisted forms can be submitted without login if the form is published. The difference is only visibility: public forms can be listed inside the product, while unlisted forms are direct-link only. At minimum, forms should support short text, long text, email, number, single select and multi select. Additional fields like dropdown, checkbox, rating and date are encouraged. tRPC must be used for type-safe APIs between frontend and backend. API documentation using OpenAPI Specification with Scalar is required.Rate limiting must be implemented for public form response submission APIs. The project should include a pricing page with dummy SaaS-style plans such as Free, Pro and Business. Real payment integration is not required. Creators should be able to create or choose themes for forms. Themes can be based on movies, anime, games, tech companies, operating systems, startups, events, communities or any other creative idea. Respondents should be able to optionally receive an email copy of their submitted response if they provide their email. Creators should be able to enable or disable email notifications when a new response is submitted. Admin or platform owner features are optional bonus features. If you add them, provide working admin credentials in the README and submission notes.
